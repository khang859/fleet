# Protocols & Navigator — Design

## Overview

Protocols are Admiral-level playbooks — structured, multi-step autonomous workflows that the Admiral invokes when a user asks for a feature to be researched and planned. The **Navigator** executes them using the fleet CLI, advancing through steps without human intervention until a defined gate. The ultimate goal is to produce a **Feature Brief** — a structured artifact with enough information for the Admiral to create well-scoped missions for implementing a feature.

---

## Glossary

| Term                   | Meaning                                                                      |
| ---------------------- | ---------------------------------------------------------------------------- |
| **Protocol**           | A named, DB-stored multi-step workflow the Admiral can invoke                |
| **Navigator**          | The autonomous process that executes a Protocol                              |
| **Protocol Execution** | A single run of a Protocol, tracked in the DB                                |
| **Feature Brief**      | The structured Cargo output of a Protocol — missions, criteria, dependencies |
| **Gate**               | A checkpoint requiring operator input before the execution can continue      |
| **Review Crew**        | A crew deployed to validate the research brief before finalization           |

## Protocols

### Storage

Protocols are stored in the database, not as markdown files. This makes them discoverable, manageable via the fleet CLI, and queryable by the Navigator at runtime.

```sql
CREATE TABLE protocols (
  id TEXT PRIMARY KEY,
  slug TEXT UNIQUE NOT NULL,         -- 'research-and-deploy'
  name TEXT NOT NULL,                -- 'Research and Deploy'
  description TEXT,                  -- shown in fleet protocols list
  help_text TEXT,                    -- detailed usage, shown in fleet protocols show
  trigger_examples TEXT,             -- JSON array of example phrases
  enabled INTEGER DEFAULT 1,
  built_in INTEGER DEFAULT 0,        -- shipped with Fleet, not deletable
  created_at DATETIME DEFAULT (datetime('now')),
  updated_at DATETIME DEFAULT (datetime('now'))
);

CREATE TABLE protocol_steps (
  id INTEGER PRIMARY KEY,
  protocol_id TEXT REFERENCES protocols(id) ON DELETE CASCADE,
  step_order INTEGER NOT NULL,
  type TEXT NOT NULL,                -- 'deploy-crew' | 'await-comms' | 'review' | 'gate' | 'complete'
  config TEXT NOT NULL,              -- JSON, step-specific params
  description TEXT,                  -- human-readable label
  UNIQUE(protocol_id, step_order)
);
```

### Fleet CLI

```bash
fleet protocols list                          # List all protocols with slug, name, status
fleet protocols show <slug>                   # Full detail: steps, help text, trigger examples
fleet protocols enable <slug>
fleet protocols disable <slug>
fleet protocols executions list               # Active and recent executions
fleet protocols executions show <id>          # Execution detail: step, status, crew IDs
fleet protocols executions update <id> --step N --status <status>
```

### Built-in Protocols

Fleet ships with one built-in Protocol pre-seeded:

**`research-and-deploy`** — Deploy a research crew to investigate the codebase and existing patterns, synthesize findings into a Feature Brief, run a review pass, then gate for operator approval before the Admiral creates missions.

## Navigator

### Role

The Navigator is an autonomous Claude Code session that drives a Protocol Execution forward. It is **not** the Admiral and **not** the First Officer — it is a dedicated executor with a narrowly scoped role: read the protocol steps, dispatch the right crews via fleet CLI, monitor comms for completion signals, and advance through each step without human input until a gate.

Same spawn pattern as the First Officer: ephemeral, fresh process per invocation, exits when its step or gate is reached.

### Workspace

```
~/.fleet/starbases/starbase-<starbase-id>/
  navigator/
    CLAUDE.md           -- Role instructions + fleet CLI reference (generated once)
    context/            -- Per-execution context files passed between Navigator spawns
      <execution-id>.md
```

Path follows the same `starbase-${starbaseId}` prefix convention as the existing FO workspace (`first-officer/`) and Admiral workspace (`admiral/`).

### CLAUDE.md

Generated by `generateNavigatorClaudeMd()` in `workspace-templates.ts`. Written once to the Navigator workspace on first spawn. Contains:

- Role identity: "You are the Navigator, not the Admiral or First Officer"
- Navigator workflow (read protocol steps first, tag all actions with execution ID, poll comms, advance steps)
- Full fleet CLI reference scoped to Navigator commands (see CLI section below)
- How to write gate, clarification, complete, and failed comms
- Max review loop iterations (3) before forcing a gate
- Cleanup rules (recall crew before exit on failure/cancel)

### Spawn

The Navigator uses the same spawn pattern as the First Officer:

```ts
spawn(
  'claude',
  [
    '--output-format',
    'stream-json',
    '--verbose',
    '--input-format',
    'stream-json',
    '--dangerously-skip-permissions',
    '--model',
    model,
    '--append-system-prompt-file',
    spFile
  ],
  { cwd: workspace, env: mergedEnv, stdio: ['pipe', 'pipe', 'pipe'] }
);
```

Initial message sent via stdin (stream-json protocol, same as FO) pointing to a context file: protocol slug, execution ID, current step, feature request, accumulated context from prior spawns.

Env vars: `FLEET_NAVIGATOR=1`, `FLEET_EXECUTION_ID=<id>`, `FLEET_STARBASE_ID=<id>`

Only one Navigator invocation is active per execution at a time. The `navigator_max_concurrent` limit (default: 2) is **global** — it caps how many distinct executions can have an active Navigator simultaneously, not how many Navigators can run for the same execution. This is consistent with the sequential gate/resume lifecycle.

### Lifecycle per Invocation

1. Run `fleet protocols show <slug>` — load protocol steps
2. Run `fleet protocols executions show <id>` — confirm current step and state
3. Execute current step (deploy crew, await comms, run review loop, etc.)
4. On step completion: run `fleet protocols executions update <id> --step N+1`
5. Continue autonomously until a gate or terminal state
6. On gate: write `gate-pending` comm to Admiral, exit cleanly
7. On complete: write `protocol-complete` comm with Feature Brief cargo ID, exit
8. On failure: recall active crew, write `protocol-failed` comm, exit

## Core Execution Flow

```
User → Admiral → spawn Navigator
  │
  ├─ Step 1: Deploy Research Crew (fleet crew deploy)
  ├─ Step 2: Poll comms until research cargo arrives
  ├─ Step 3: Deploy Review Crew with brief + original feature request
  ├─ Step 4: Poll comms for review-pass or review-fail
  │     ├─ review-fail → revise brief, re-deploy Review Crew (max 3 iterations)
  │     └─ review-pass → finalize Feature Brief as Cargo
  └─ Step 5: [GATE] write gate-pending comm to Admiral, exit

Admiral surfaces gate to operator
  ├─ Approved → Admiral spawns new Navigator with "resume from step 6, gate approved"
  ├─ Adjusted → Admiral amends context, spawns Navigator with amendments
  └─ Rejected → execution cancelled, active crew recalled
```

## Feature Brief (Cargo)

The Feature Brief is the structured Cargo output. It contains:

- **Feature summary** — what was requested and what was found
- **Proposed missions** — one per discrete unit of work, each with:
  - Sector assignment
  - Mission description
  - Acceptance criteria
  - Estimated complexity
- **Mission sequencing** — which missions depend on which
- **Risks and unknowns** — open questions surfaced during research
- **Review notes** — what the Review Crew caught and what was revised

The Admiral reads this to create missions. Each created mission is tagged with `protocol_execution_id`.

## Protocol Executions

```sql
CREATE TABLE protocol_executions (
  id TEXT PRIMARY KEY,
  protocol_id TEXT REFERENCES protocols(id),
  status TEXT NOT NULL DEFAULT 'running',
    -- 'running' | 'gate-pending' | 'complete' | 'failed' | 'cancelled' | 'gate-expired'
  current_step INTEGER NOT NULL DEFAULT 1,
  feature_request TEXT NOT NULL,     -- original user ask
  context TEXT,                      -- JSON, accumulated context between Navigator spawns
  active_crew_ids TEXT,              -- JSON array, for cleanup
  created_at DATETIME DEFAULT (datetime('now')),
  updated_at DATETIME DEFAULT (datetime('now'))
);
```

### Missions FK

```sql
ALTER TABLE missions ADD COLUMN protocol_execution_id TEXT REFERENCES protocol_executions(id);
```

### Comms FK

```sql
-- Migration 12 (see Migrations section)
ALTER TABLE comms ADD COLUMN execution_id TEXT REFERENCES protocol_executions(id);
```

All comms written by the Navigator or its crews include `execution_id`. The Navigator polls `fleet comms inbox --execution <id> --unread` which filters on this column. This scopes comms to the correct execution when multiple Protocols run concurrently.

**`CommsService` amendments required:**

1. **`SendOpts` type** — add `execution_id?: string` field
2. **`INSERT` statement** — include `execution_id` in the column list, populated from `opts.execution_id ?? null`
3. **Dedup exclusion** — update the dedup check from `opts.from !== 'admiral'` to also skip `'navigator'`, preventing identical comms from different executions being coalesced

Without (1) and (2), `execution_id` will always be NULL on Navigator-written comms, breaking the `--execution` filter entirely.

### Feature Brief Cargo Schema

A Feature Brief is registered as a Cargo record with:

- `crew_id` — the final Navigator invocation's process ID
- `mission_id` — NULL (pre-mission artifact)
- `sector_id` — `'global'` sentinel value (the `cargo.sector_id` column is NOT NULL REFERENCES sectors, so migration 11 seeds a `sectors` row with `id = 'global'` to satisfy the FK)
- `protocol_execution_id` — the execution that produced it
- `type` — `'feature-brief'`
- `manifest` — JSON encoding the Feature Brief fields (summary, proposed missions, sequencing, risks, review notes)

```sql
-- Migration 12 (see Migrations section)
ALTER TABLE cargo ADD COLUMN protocol_execution_id TEXT REFERENCES protocol_executions(id);
```

## Gate Handling

1. Navigator writes `gate-pending` comm to Admiral with:
   - Execution ID
   - Step number
   - Feature Brief cargo ID (or clarification question)
   - Decision context
2. Admiral surfaces gate in Comms feed
3. Operator responds via Admiral:
   - **Approved** → Admiral spawns new Navigator with gate response in context, execution resumes
   - **Adjusted** → Admiral amends Feature Brief, spawns Navigator with amendments
   - **Rejected** → Admiral cancels execution, Navigator recalled, clean exit

### Gate Expiry

Sentinel detects `gate-pending` executions older than `navigator_gate_expiry` config (default: 24h). Marks execution `gate-expired`, writes summary comm to Admiral.

---

## Error Handling

### Individual crew failures

FO handles these as normal (retry/escalate). When a crew tagged with a `protocol_execution_id` exhausts FO retries, Sentinel also triggers the Navigator for that execution with a `crew-failed` event.

Navigator decides:

- Can it recover? (redeploy, use partial cargo, skip optional step)
- Cannot recover → recall all active crew, write `protocol-failed` comm, exit

### Navigator crash

If the Navigator process itself crashes, Sentinel writes a fallback `protocol-failed` comm (same pattern as FO's fallback escalation memo).

## Clarifications

**Two tiers:**

| Type                                                                     | Who resolves | Mechanism                                                                |
| ------------------------------------------------------------------------ | ------------ | ------------------------------------------------------------------------ |
| Ambiguous sector choice, inferable context                               | Navigator    | Picks most likely option, notes decision in execution context, continues |
| Ambiguous feature request, operator preference, conflicting requirements | Human        | Navigator writes `clarification-needed` gate comm, pauses execution      |

Clarification gates behave identically to regular gates — operator responds via Admiral, Navigator resumes with the answer baked into context.

New comms types added by this feature:

| Type                   | From        | To        | Meaning                                      |
| ---------------------- | ----------- | --------- | -------------------------------------------- |
| `gate-pending`         | navigator   | admiral   | Execution paused, awaiting operator decision |
| `protocol-complete`    | navigator   | admiral   | Execution finished, Feature Brief ready      |
| `protocol-failed`      | navigator   | admiral   | Execution failed, details attached           |
| `clarification-needed` | navigator   | admiral   | Ambiguous input, needs operator response     |
| `review-pass`          | review-crew | navigator | Brief approved by Review Crew                |
| `review-fail`          | review-crew | navigator | Brief has issues, notes attached             |

## Cleanup

| Terminal status | Who cleans up                              | What                                            |
| --------------- | ------------------------------------------ | ----------------------------------------------- |
| `complete`      | Navigator on exit                          | Temp files, execution context file              |
| `failed`        | Navigator on exit (or Sentinel if crashed) | Temp files, recall active crew                  |
| `cancelled`     | Admiral on operator request                | Recall active crew, mark execution cancelled    |
| `gate-expired`  | Sentinel sweep                             | Mark execution gate-expired, write summary comm |

Execution records, comms, and Cargo are **retained** — swept by RetentionService after the configured window. Cargo lives longer as it has standalone value.

`RetentionService` must be amended:

- Add `protocol_executions` to the `TABLES` constant for both sweep and stats
- Add `protocols` and `protocol_steps` to `TABLES` for **stats only** (row counts) — they are configuration, not operational data, and are never deleted by retention
- New config key: `protocol_executions_retention_days` (default: 30)

## Navigator Fleet CLI — Agent-Optimized Layer

Because the Navigator runs Haiku by default (`navigator_model` config, default `claude-haiku-4-5`), the fleet CLI detects `FLEET_NAVIGATOR=1` and enables structured, hint-rich output for all commands.

### Response envelope

```json
{ "ok": true, "data": { ... }, "hint": "suggested next command" }
{ "ok": false, "error": "human-readable reason", "hint": "corrective action" }
```

### Next-step hints

Every successful command includes a `hint` guiding the Navigator toward the correct follow-up:

```json
{
  "ok": true,
  "data": { "crewId": "crew_789", "status": "deploying" },
  "hint": "Crew deploying. Poll with `fleet comms inbox --execution exec_456 --unread` to detect completion"
}
```

### Actionable errors

```json
{
  "ok": false,
  "error": "crew 'abc123' not found",
  "hint": "Run `fleet crew list --execution exec_456` to see active crew for this execution"
}
```

### Pre-flight validation

Irreversible commands (`crew recall`, `crew deploy`) validate all inputs upfront — sector exists, mission is correct state, execution ID is valid — and return structured errors before executing anything.

### Execution-scoped commands

Every Navigator command accepts `--execution <id>`. Attempting to act on a crew or mission that does not belong to the current execution returns a clear error rather than silently proceeding.

**`fleet comms inbox --execution <id> --unread`** filters the `comms` table on `execution_id = <id> AND read = 0`. This is a new flag added to the existing `fleet comms inbox` command — not a new subcommand. Returns structured JSON array of unread comms for this execution, each with `type`, `from_crew`, `payload`, and `id` for acknowledgement.

### Protocol step guard

`fleet protocols executions update --step N` validates that `current_step` in the DB equals N-1 before advancing. If the Navigator crashes mid-step before calling the update, `current_step` remains at N-1, so a resumed Navigator re-executes the step from scratch (idempotent by design — recalling and redeploying crew is safe). Prevents step skipping if the Navigator loses track.

### Migration 11 — Protocol tables + global sector seed

Creates the three new tables and seeds the `'global'` sector sentinel row required by Feature Brief cargo.

```sql

  id TEXT PRIMARY KEY,

  type TEXT NOT NULL,

  id TEXT PRIMARY KEY,
  protocol_id TEXT REFERENCES protocols(id),
```

## Sentinel Additions

### Navigator sweep

Added alongside the existing `firstOfficerSweep()` in `runSweep()`:

1. **Crew-failed fan-out** — when FO writes an auto-escalation comm (type `memo`) for a mission that has a `protocol_execution_id`, Sentinel's Navigator sweep detects it via the following join:

   ```sql
   SELECT c.execution_id
   FROM comms c
   JOIN missions m ON c.mission_id = m.id
   WHERE c.type = 'memo'
     AND m.protocol_execution_id IS NOT NULL
     AND c.read = 0
   ```

   This triggers a Navigator invocation with event type `crew-failed` for that execution. FO's existing escalation path is unchanged — the Navigator sweep is an additional fan-out, not a replacement.

2. **Gate expiry sweep** — detect `gate-pending` executions where `updated_at` is older than `navigator_gate_expiry` seconds → mark `gate-expired`, write summary comm to Admiral.

3. **In-memory dedup map** — running Navigator processes keyed by `execution_id` (same pattern as FO's `crew_id:mission_id` map). Only one Navigator per execution at a time.

4. **Crash fallback** — if a Navigator process exits non-zero, Sentinel writes a `protocol-failed` comm (same pattern as FO's fallback escalation memo).
